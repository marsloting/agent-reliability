# 跨域重读

**过期的记忆不报错。agent 不知道自己已经不知道了——你也不知道，直到下游悄悄把它的产出扔掉。**

## 哪里出问题

agent 在 A 域干活，然后要往 B 域产出——另一个工作流的收件文件、一个带 schema 校验的文档、一个有格式要求的协议头。它"记得"B 的格式：本次会话早些时候见过，或者上次碰 B 时的印象。它产出得非常自信。零报错。

但 B 的契约在那段记忆形成之后改了。多了一个必填字段。头部格式升级了。路径挪了。

最狠的地方：**产出这一侧的全链路没有任何失败信号。** 记忆自洽，产出按旧契约衡量格式完美，写入成功。失败只存在于消费端——下游静默拒收、错误解析、或直接丢弃。下游要也是个自动化环节，那就彻底没人看见。这会和[虚假成功](corrupt-success.zh.md)叠加：产出方 agent 还会照实报"done"——按它那份过期标准，确实 done 了。

这和 [context rot](https://research.trychroma.com/context-rot) 不是一回事——那是单一上下文变长后的性能衰减。跨域过期不需要长上下文：五分钟前的记忆，对四分钟前改掉的契约，照样全军覆没。rot 是逐渐变笨，过期记忆是躺着说谎。

## 我在生产里踩到的

多条 agent 工作流靠文件交接，交接格式有版本。一个 agent 凭记忆里的条目格式，往姊妹工作流的收件文件里写了一批条目。格式其实已经升级——加了一行必须的机器可读元数据。它写的每一条都被当成旧版条目，从自动化管线里漏了出去。

没有异常。没有警告。只是下游的统计数字偏低，靠人盯着觉得"这数不对"才翻出一批写得很好、但整体过期的条目。agent 的知识在形成那一刻没有错，它只是过期了——而过期不发事件。

同类问题第三次咬我的时候，模式已经藏不住了：每一次出事，都是**跨着边界、凭记忆产出**。

## 防御

1. **把所有外部契约的记忆当成一个没有 TTL、没有失效通知的缓存。** schema、协议、API 形状、文件格式、目录结构——只要在当前工作域之外，对它的记忆一律按已过期处理。
2. **在边界上重读契约，再产出。** 不是"使劲回忆"，是真的打开目标域当前的 schema/规范/README。读一次几秒钟；静默错配的代价是一场 debug，外加下游已经丢掉的所有东西。
3. **往另一个域批量产出 → 走[首件检验](first-article-inspection.zh.md)。** 第一件按"刚读到的"契约做端到端验证（消费端真的收吗？），过了再产剩下的。
4. **同类错配第二次出现 → 停止逐个修，做契约全量 diff。** 第二次意味着你对这条边界的认知是系统性过期，不是偶尔倒霉。把整份契约重读一遍，和你的假设逐项 diff——在实例层面打补丁，只是把第三次事故排上日程。

## 规则——可直接粘进 `CLAUDE.md` / `AGENTS.md`

> 当前域之外的任何契约（schema / 协议 / API / 文件格式），对它的记忆一律视为已过期缓存。跨域产出前，必须从源头重读目标域当前契约——禁止凭记忆产出。批量产出先用首件过消费端验证，再产其余。
